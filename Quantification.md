## 一、系统总体架构

### 1.1 核心数据流

```
爬虫层（行情 / 舆情 / 研报 / 产业链）
        ↓
数据清洗治理层（P0）
        ↓
统一数据仓库（MySQL / TSDB / Redis）
        ↓
因子计算引擎（技术 / 情绪 / 机构 / 产业）
        ↓

        ↓
回测引擎
        ↓
实盘交易系统
        ↓
AI Agent 分析层
        ↓
Web 可视化平台
```

### 1.3 MVP 闭环定义

> **MVP 目标**：优先打通「日线数据 → 清洗 / 复权 → 技术因子 → 单策略回测 → 报告输出」闭环，先验证数据可信度和策略评估能力，再扩展舆情、研报、实盘和 Agent。

**MVP 范围**

- 日线 K 线数据清洗、去重、缺失检查、前复权。
- A 股交易日历统一。
- 技术因子库 MVP：MA、MACD、RSI、ATR、动量、均线突破。
- 单策略回测 MVP：T+1、手续费、印花税、滑点、涨跌停和停牌处理。
- 输出回测报告：收益曲线、最大回撤、夏普比率、胜率、盈亏比、交易明细。

**MVP 暂不包含**

- 自动实盘下单。
- Tick 级高频回测。
- 多 Agent 自动决策闭环。
- 复杂组合优化和机器学习模型训练。

**MVP 验收标准**

- 可对不少于 100 只 A 股运行最近 3 年日线回测。
- 回测结果可重复，固定输入数据与参数时输出一致。
- 每次回测可追溯使用的数据版本、因子版本、策略版本和参数。
- 对低质量数据可主动告警或拒绝进入因子 / 回测流程。

### 1.4 目标工程结构

```text
quant_system/
  data_source/      # 数据采集：行情、舆情、研报、产业链
  pipeline/         # 清洗、标准化、复权、质量校验
  storage/          # JSON / MySQL / TSDB / Redis 存储适配
  factors/          # 因子计算与因子版本管理
  strategy/         # 选股、择时、调仓、组合规则
  backtest/         # 回测引擎、成交撮合、绩效分析
  risk/             # 回测与实盘共用风控规则
  agent/            # 数据巡检、策略分析、选股解释、舆情解读
  web/              # 行情、因子、舆情、回测、实盘可视化
  tasks/            # 定时任务与批处理入口
```

**模块边界**

- `data_source` 只负责采集原始数据，不直接做策略判断。
- `pipeline` 负责清洗、标准化、复权和质量评分。
- `factors` 只消费标准化后的可信数据，不直接调用爬虫。
- `strategy` 只消费行情、因子和基础面数据，不处理原始数据清洗。
- `backtest` 不依赖实盘交易接口，避免回测与实盘逻辑耦合。
- `risk` 同时服务回测和实盘，保证两端风控规则一致。

### 1.5 阶段能力矩阵

| 阶段 | 核心目标 | 支持功能 | 可回答的问题 | 预测能力 |
|------|----------|----------|--------------|----------|
| Phase 0 数据基建 | 数据可用、可信、可计算 | 日线 / 分钟线采集、清洗、复权、交易日历、质量评分、补爬 | 这只股票的数据是否完整可信？今天是否能参与分析？ | 不做走势预测，只提供可信输入 |
| Phase 1 多源数据 | 扩展信息维度 | 舆情、研报、产业链、财报、指数、龙虎榜等数据接入 | 这只股票是否有事件、情绪或机构预期变化？ | 可形成事件和情绪观察，不输出交易级预测 |
| Phase 2 因子系统 | 把数据转成可量化信号 | 技术、情绪、机构、产业因子计算与排名 | 这只股票当前动量、趋势、情绪、估值是否占优？ | 具备初级方向判断能力，但尚未验证胜率 |
| Phase 3 回测系统 | 验证信号有效性 | 单策略 / 多因子回测、成交模拟、绩效分析、风险过滤 | 某类信号过去是否有效？胜率、回撤、收益风险比如何？ | 具备可验证的走势预测能力，可输出概率化方向判断 |
| Phase 4 实盘交易 | 小规模实盘验证 | 实时行情、交易接口、风控、持仓、订单日志 | 回测有效的信号在真实成交中是否仍有效？ | 具备实盘监控下的短周期预测和执行能力 |
| Phase 5 AI Agent | 自动分析和解释 | 数据巡检、策略诊断、选股解释、舆情解读、参数建议 | 为什么看多 / 看空？哪些证据支持？策略为什么失效？ | 具备解释型预测能力，默认不自动下单 |
| Phase 6 Web 可视化 | 统一操作入口 | 看板、因子排名、预测结果、回测报告、实盘监控 | 哪些股票值得关注？预测、证据、风险和历史表现如何？ | 提供可读、可追踪、可对比的预测结果展示 |

**关键结论**

- Phase 0 只解决“数据能不能信”，不应该输出走势判断。
- Phase 2 可以输出“信号强弱”和“趋势倾向”，但不能直接视为有效预测。
- Phase 3 完成后，系统才具备最低可接受的股票走势预测能力，因为预测信号经过历史回测验证。
- Phase 4 完成后，系统才具备实盘可用的预测闭环，但仍需人工授权和风控拦截。
- Phase 5 让预测具备解释能力，输出“方向 + 概率 + 证据 + 风险”，而不是只给涨跌结论。

## 二、阶段拆分

### Phase 0：数据基建（P0）

> **目标**：让所有数据「可用、可信、可计算」——必须优先完成。

#### 2.1 数据清洗与标准化

**功能模块**

- K 线异常数据过滤（涨跌停异常、停牌、0 成交量）
- 去重（分钟级 / 日级）
- 缺失数据补齐
- 股票代码统一（6 位标准）
- 交易日历统一（A 股）

**标准 K 线结构**

```json
{
  "ts": "",
  "code": "",
  "open": 0,
  "high": 0,
  "low": 0,
  "close": 0,
  "volume": 0,
  "amount": 0,
  "adj_factor": 1
}
```

**字段约束**

- `ts`：ISO 8601 时间字符串，默认使用 Asia/Shanghai 时区。
- `code`：6 位 A 股股票代码，不包含交易所后缀。
- `open/high/low/close`：单位为人民币元，保留原始精度。
- `volume`：成交量，单位为股。
- `amount`：成交额，单位为人民币元。
- `adj_factor`：复权因子，未复权时默认 `1`。

#### 2.2 复权系统

- 前复权
- 后复权
- 分红送股修复
- `adj_factor` 版本管理
- 复权公式、数据来源和更新时间必须可追溯

#### 2.3 数据校验与质量监控

- 跨源一致性校验（东财 vs 新浪）
- 延迟 / 缺失告警
- 数据版本与采集时间戳
- 低质量数据隔离，禁止直接进入因子与回测

**数据质量评分结构**

```json
{
  "code": "000001",
  "trade_date": "2026-07-01",
  "missing_rate": 0,
  "duplicate_count": 0,
  "cross_source_diff": 0.001,
  "quality_score": 98,
  "status": "ok",
  "checked_at": ""
}
```

**质量规则**

- `quality_score >= 90`：可进入因子与回测。
- `70 <= quality_score < 90`：可用于展示，但回测需显式允许。
- `quality_score < 70`：隔离并触发告警 / 补爬。
- 跨源价格差异超过阈值时，以数据更新时间、成交量完整性和来源优先级综合判定。

#### 2.4 统一数据仓库

- MySQL：元数据、财报、研报
- TSDB：分钟 / Tick 时序
- Redis：盘中实时缓存（`live:stock:{code}`）

#### 2.5 数据源契约

| 数据 | 优先来源 | 频率 | 目标存储 | 降级策略 |
|------|----------|------|----------|----------|
| 日 K | 东财 | 每日收盘后 | JSON / TSDB | 新浪兜底，失败则告警 |
| 分钟线 | 东财 | 盘中定时 | Redis / JSON | 缺失标记，不参与盘中信号 |
| 指数数据 | 东财 | 每日 / 盘中 | JSON / TSDB | 新浪兜底 |
| 财报 | 待定 | 季度 | MySQL | 人工补录 / 延迟更新 |
| 舆情 | 雪球 | 定时采集 | MySQL / JSON | 跳过并记录失败 |
| 研报 | 待定 | 每日 | MySQL | 延迟更新 |

**通用采集要求**

- 所有原始数据必须保留 `source`、`fetched_at`、`raw_version`。
- 清洗后的数据必须保留 `data_version`。
- 多源降级必须在日志和质量报告中记录，不能静默覆盖。

#### 2.6 基础数据补齐

- PE / PB / ROE
- 财报数据
- 龙虎榜
- 解禁 / 分红 / 业绩预告
- 指数数据（沪深 300 / 中证 500）
- 利率数据

### Phase 2：因子系统（P1–P2）

#### 4.1 技术因子

- MA / MACD / RSI / KDJ
- ATR / 波动率
- 动量因子
- 均线突破

#### 4.2 情绪因子

- 多空情绪比
- 热度变化率
- 情绪背离

#### 4.3 机构因子

- 目标价空间
- 评级变化
- 机构覆盖度

#### 4.4 产业因子

- 涨价强度
- 缺货指数
- 事件驱动评分

#### 4.5 因子标准输出

```json
{
  "code": "",
  "trade_date": "",
  "factors": {
    "ma20_bias": 0,
    "rsi14": 0,
    "sentiment_score": 0,
    "target_price_premium": 0
  },
  "updated_at": ""
}
```

#### 4.6 因子计算规范

- 因子输入只能来自清洗、复权、质量校验后的标准数据。
- 因子计算不得使用未来数据，所有窗口只能包含当前交易日及之前的数据。
- 每个因子必须有名称、公式说明、参数、版本、适用频率和缺失值处理规则。
- 公式或参数默认值变更时，必须生成新的因子版本。
- 上市不足 N 天、停牌、成交额过低等场景需要明确输出 `null`、跳过或降权。

**因子明细输出结构**

```json
{
  "code": "000001",
  "trade_date": "2026-07-01",
  "factor_name": "rsi14",
  "factor_value": 62.5,
  "factor_version": "1.0.0",
  "data_version": "daily_kline_20260701",
  "updated_at": ""
}
```

#### 4.7 初级走势信号输出

Phase 2 完成后，可以输出技术层面的初级走势信号，但该信号只代表“因子倾向”，不能直接等同于可交易预测。

```json
{
  "code": "000001",
  "trade_date": "2026-07-01",
  "horizon": "5d",
  "signal": "bullish",
  "signal_score": 72,
  "drivers": ["ma20_breakout", "rsi14_neutral_up", "volume_expansion"],
  "limitations": ["not_backtested", "technical_only"],
  "updated_at": ""
}
```

**信号解释**

- `signal`：`bullish`、`bearish`、`neutral` 三类方向。
- `signal_score`：0-100 分，只表示当前因子强度。
- `horizon`：预测观察周期，如 `1d`、`5d`、`20d`。
- `limitations`：必须标明是否经过回测、是否只包含技术因子。

### Phase 3：回测系统（P2）

#### 5.1 回测引擎能力

- T+1 交易规则
- 手续费 / 滑点
- 涨跌停限制
- 停牌处理
- 回测与实盘共用风控模块

#### 5.2 策略类型

- 技术策略
- 情绪策略
- 机构策略
- 产业链策略
- 多因子策略

#### 5.3 绩效指标

- 年化收益
- 最大回撤
- 夏普比率
- 胜率
- 盈亏比

#### 5.4 回测成交规则

**默认成交假设**

- 买入价：信号日后第一个可交易日开盘价。
- 卖出价：触发卖出信号后第一个可交易日开盘价。
- T+1：当日买入持仓不可当日卖出。
- 涨停：买入日涨停且无法成交时，订单取消或顺延，需由策略参数指定。
- 跌停：卖出日跌停且无法成交时，继续持有并记录未成交原因。
- 停牌：停牌期间不可买卖，价格沿用最近可交易日用于净值估算。

**交易成本默认值**

- 佣金：万分之三，最低 5 元。
- 印花税：卖出单边千分之一。
- 滑点：默认 5bp，可按策略覆盖。

#### 5.5 组合与风控规则

- 单票最大仓位：默认 10%。
- 单行业 / 单题材最大暴露：默认 30%。
- 最低流动性门槛：近 20 日日均成交额不低于指定阈值。
- 排除 ST、退市整理、新股上市不足 N 日标的。
- 最大回撤熔断：触发后停止开新仓或降仓。
- 黑名单机制：支持手动和规则生成。

#### 5.6 可验证走势预测输出

Phase 3 完成后，系统开始具备“可验证”的走势预测能力。这里的预测不是保证涨跌，而是基于历史统计和当前因子状态输出概率、置信度、收益风险比和失效条件。

```json
{
  "code": "000001",
  "trade_date": "2026-07-01",
  "horizon": "5d",
  "direction": "up",
  "probability": 0.61,
  "confidence": "medium",
  "expected_return": 0.028,
  "max_expected_drawdown": 0.045,
  "strategy_version": "trend_breakout_v1.0.0",
  "factor_versions": {
    "ma20_bias": "1.0.0",
    "rsi14": "1.0.0"
  },
  "evidence": {
    "backtest_win_rate": 0.57,
    "backtest_profit_loss_ratio": 1.35,
    "sample_count": 420
  },
  "risk_flags": ["earnings_nearby", "sector_high_volatility"],
  "updated_at": ""
}
```

**最低验收标准**

- 至少支持 `1d`、`5d`、`20d` 三类观察周期之一，MVP 优先 `5d`。
- 每个预测必须关联策略版本、因子版本和数据版本。
- 预测必须有历史样本数量，样本过少时只能输出 `low_confidence`。
- 必须同时输出方向、概率、置信度、预期收益、预期回撤和风险标记。
- 不允许输出“必涨”“必跌”等确定性结论。

### Phase 4：实盘交易系统（P2）

## 三、股票走势预测能力路径

### 3.1 预测能力分级

| 等级 | 达成阶段 | 能力描述 | 输出形式 | 是否可用于交易决策 |
|------|----------|----------|----------|--------------------|
| L0 | Phase 0 | 数据质量判断 | 数据完整性、质量分、异常标记 | 否 |
| L1 | Phase 2 | 因子倾向判断 | 看多 / 看空 / 中性信号、因子分数 | 否，只能作为观察 |
| L2 | Phase 3 | 回测验证预测 | 方向、概率、置信度、收益风险比、历史胜率 | 可用于模拟交易和人工辅助 |
| L3 | Phase 4 | 实盘验证预测 | 实时信号、订单可执行性、风控结果 | 可用于小资金实盘，但需授权 |
| L4 | Phase 5 | 解释型预测 | 预测结论、证据链、风险、失效条件、策略诊断 | 可用于投研决策，不默认自动交易 |
| L5 | Phase 6 | 产品化预测 | 看板、排名、预警、回测对比、实盘跟踪 | 可作为统一操作入口 |

### 3.2 预测闭环定义

股票走势预测必须形成以下闭环：

```text
可信数据
  → 因子信号
  → 策略规则
  → 历史回测
  → 当前预测
  → 风险过滤
  → 结果展示 / 模拟交易 / 实盘验证
  → 预测复盘
```

若缺少“历史回测”和“预测复盘”，只能称为信号提示，不能称为完整预测能力。

### 3.3 预测输出边界

**允许输出**

- 未来 `1d / 5d / 20d` 的方向倾向。
- 上涨 / 下跌概率。
- 置信度：`low`、`medium`、`high`。
- 预期收益区间和预期回撤。
- 关键驱动因子和风险因素。
- 策略历史表现和样本数量。

**禁止输出**

- 确定性涨跌结论。
- 没有数据版本、策略版本和回测证据的预测。
- 绕过风控的买卖建议。
- 由 Agent 直接触发的默认实盘下单。

### 3.4 到哪一步能完成走势预测

- **Phase 2 完成后**：可以完成“初级走势信号”，例如趋势偏多、偏空、震荡，但还不能判断该信号历史上是否赚钱。
- **Phase 3 完成后**：可以完成“可验证走势预测”，这是系统首次达到可接受的预测能力门槛。预测结果应包含方向、概率、置信度、历史胜率、收益风险比和风险标记。
- **Phase 4 完成后**：可以完成“可执行走势预测”，即预测结果能接入实时行情、交易约束和风控，但仍需要人工授权。
- **Phase 5 完成后**：可以完成“可解释走势预测”，即系统不仅输出预测，还能说明为什么、依据什么、什么情况下预测失效。
- **Phase 6 完成后**：可以完成“产品化走势预测”，在 Web 看板中展示股票池、预测排名、证据链、回测表现和实盘跟踪。

**近期最小目标**：完成 Phase 3 的单策略回测 MVP 后，即可开始对股票输出 `5d` 维度的可验证走势预测。

## 四、通用约束与风险声明

### 4.1 工程约束

- 所有关键流程必须可重跑、可追溯、可审计。
- 策略、因子、数据版本需要在回测报告和实盘日志中记录。
- 任何数据源失败、降级、补爬、字段异常都必须进入日志。
- 回测和实盘应尽量共享成交约束、风控规则和标的过滤规则。

### 4.2 投资与合规风险

- 系统输出不构成投资建议，仅作为研究和辅助决策工具。
- 回测收益不代表未来收益。
- 数据源异常、延迟或字段变更可能导致信号失真。
- 实盘交易必须由用户主动授权，并保留人工干预和熔断机制。
- 涉及第三方数据采集时，需要遵守对应平台的服务条款和数据使用边界。
