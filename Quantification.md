## 一、系统总体架构

### 1.1 核心数据流

```
爬虫层（行情 / 舆情 / 研报 / 产业链）
        ↓
数据清洗治理层（P0）
        ↓
统一数据仓库（MySQL / TSDB / Redis）
        ↓
因子计算引擎（技术 / 情绪 / 机构 / 产业）
        ↓
策略层（选股 / 择时 / 风控）
        ↓
回测引擎
        ↓
实盘交易系统
        ↓
AI Agent 分析层
        ↓
Web 可视化平台
```

### 1.4 目标工程结构

```text
quant_system/
  data_source/      # 数据采集：行情、舆情、研报、产业链
  pipeline/         # 清洗、标准化、复权、质量校验
  storage/          # JSON / MySQL / TSDB / Redis 存储适配
  factors/          # 因子计算与因子版本管理
  strategy/         # 选股、择时、调仓、组合规则
  backtest/         # 回测引擎、成交撮合、绩效分析
  prediction/       # 可验证走势预测、概率输出、预测复盘
  replay/           # 历史视角滚动推演、无未来函数复盘
  attribution/      # 每日涨跌归因、上涨逻辑破坏判断、后续观察位
  evaluation/       # 预测、候选、决策的后验收益复盘
  contracts/        # Universe、Signal、Target、Risk、Execution、Analyzer 标准契约
  selector/         # 上涨候选池筛选、排序、排除原因
  impact/           # 实际影响数据：业绩、估值、解禁、材料/产品价格
  decision/         # 单股指导性操作建议、仓位建议、失效条件
  risk/             # 回测与实盘共用风控规则
  portfolio/        # 持仓、现金、净值、敞口、再平衡
  trading/          # 模拟交易、实盘接口、订单、成交、账户同步
  agent/            # 数据巡检、策略分析、选股解释、舆情解读
  web/              # 行情、因子、舆情、回测、实盘可视化
  tasks/            # 定时任务与批处理入口
  monitoring/       # 任务监控、告警、审计、权限、配置管理
```

**模块边界**

- `data_source` 只负责采集原始数据，不直接做策略判断。
- `pipeline` 负责清洗、标准化、复权和质量评分。
- `market_scope` 负责区分研究标的与参考标的；科创板个股默认不进入重型研究链路。
- `factors` 只消费标准化后的可信数据，不直接调用爬虫。
- `strategy` 只消费行情、因子和基础面数据，不处理原始数据清洗。
- `backtest` 不依赖实盘交易接口，避免回测与实盘逻辑耦合。
- `risk` 同时服务回测和实盘，保证两端风控规则一致。
- `prediction` 只能基于已通过回测验证的策略或因子组合输出概率化预测。
- `replay` 用历史时点数据逐日推演下一交易日，真实收益只用于事后复盘，并沉淀技术、大盘、资金、公告/业绩、估值、产业价格事件根因、未命中原因和阈值优化建议。
- `attribution` 对比最近两个交易日，解释昨日上涨与今日下跌的主因，输出量价、趋势、大盘、资金、事件、replay 根因和上涨逻辑是否被破坏，不直接生成交易动作。
- `evaluation` 只做后验复盘，不生成买卖建议；用于校准预测、候选和决策阈值。
- `contracts` 只做标准对象定义和现有模块输出适配，不改写各模块内部算法。
- `selector` 汇总预测、因子、趋势、回测、实际影响和风险，输出上涨候选排名，不直接生成交易动作。
- `impact` 将增强数据中的财报、估值、解禁、产业价格描述转成可评分事件，必须保留来源与限制。
- `decision` 汇总预测、因子、回测、Agent、质量和持仓上下文，只输出指导性建议，不直接下单。
- `portfolio` 汇总账户、持仓、目标仓位和风险告警，不直接下单。
- `trading` 只执行已通过风控的订单，不能绕过 `risk`。
- `monitoring` 负责采集、回测、预测、交易和 Agent 行为的运行状态与审计记录。

### 1.5 阶段能力矩阵

| 阶段 | 核心目标 | 支持功能 | 可回答的问题 | 预测能力 |
|------|----------|----------|--------------|----------|
| Phase 0 数据基建 | 数据可用、可信、可计算 | 日线 / 分钟线采集、清洗、复权、交易日历、质量评分、补爬 | 这只股票的数据是否完整可信？今天是否能参与分析？ | 不做走势预测，只提供可信输入 |
| Phase 1 多源数据 | 扩展信息维度 | 舆情、研报、产业链、财报、指数、龙虎榜等数据接入 | 这只股票是否有事件、情绪或机构预期变化？ | 可形成事件和情绪观察，不输出交易级预测 |
| Phase 2 因子系统 | 把数据转成可量化信号 | 技术、情绪、机构、产业因子计算与排名 | 这只股票当前动量、趋势、情绪、估值是否占优？ | 具备初级方向判断能力，但尚未验证胜率 |
| Phase 3 回测系统 | 验证信号有效性 | 单策略 / 多因子回测、成交模拟、绩效分析、风险过滤 | 某类信号过去是否有效？胜率、回撤、收益风险比如何？ | 具备可验证的走势预测能力，可输出概率化方向判断 |
| Phase 4 实盘交易 | 小规模实盘验证 | 实时行情、交易接口、风控、持仓、订单日志 | 回测有效的信号在真实成交中是否仍有效？ | 具备实盘监控下的短周期预测和执行能力 |
| Phase 5 AI Agent | 自动分析和解释 | 数据巡检、策略诊断、选股解释、舆情解读、参数建议 | 为什么看多 / 看空？哪些证据支持？策略为什么失效？ | 具备解释型预测能力，默认不自动下单 |
| Phase 6 Web 可视化 | 统一操作入口 | 看板、因子排名、预测结果、回测报告、实盘监控 | 哪些股票值得关注？预测、证据、风险和历史表现如何？ | 提供可读、可追踪、可对比的预测结果展示 |

**关键结论**

- Phase 0 只解决“数据能不能信”，不应该输出走势判断。
- Phase 2 可以输出“信号强弱”和“趋势倾向”，但不能直接视为有效预测。
- Phase 3 完成后，系统才具备最低可接受的股票走势预测能力，因为预测信号经过历史回测验证。
- Phase 4 完成后，系统才具备实盘可用的预测闭环，但仍需人工授权和风控拦截。
- Phase 5 让预测具备解释能力，输出“方向 + 概率 + 证据 + 风险”，而不是只给涨跌结论。

## 二、阶段拆分

### Phase 0：数据基建（P0）

> **目标**：让所有数据「可用、可信、可计算」——必须优先完成。

#### 2.1 数据清洗与标准化

**功能模块**

- K 线异常数据过滤（涨跌停异常、停牌、0 成交量）
- 去重（分钟级 / 日级）
- 缺失数据补齐
- 股票代码统一（6 位标准）
- 交易日历统一（A 股）

**标准 K 线结构**

```json
{
  "ts": "",
  "code": "",
  "open": 0,
  "high": 0,
  "low": 0,
  "close": 0,
  "volume": 0,
  "amount": 0,
  "adj_factor": 1
}
```

**字段约束**

- `ts`：ISO 8601 时间字符串，默认使用 Asia/Shanghai 时区。
- `code`：6 位 A 股股票代码，不包含交易所后缀。
- `open/high/low/close`：单位为人民币元，保留原始精度。
- `volume`：成交量，单位为股。
- `amount`：成交额，单位为人民币元。
- `adj_factor`：复权因子，未复权时默认 `1`。

#### 2.2 复权系统

- 前复权
- 后复权
- 分红送股修复
- `adj_factor` 版本管理
- 复权公式、数据来源和更新时间必须可追溯

#### 2.3 数据校验与质量监控

- 跨源一致性校验（东财 vs 新浪）
- 延迟 / 缺失告警
- 数据版本与采集时间戳
- 低质量数据隔离，禁止直接进入因子与回测

**数据质量评分结构**

```json
{
  "code": "000001",
  "trade_date": "2026-07-01",
  "missing_rate": 0,
  "duplicate_count": 0,
  "cross_source_diff": 0.001,
  "quality_score": 98,
  "status": "ok",
  "checked_at": ""
}
```

**质量规则**

- `quality_score >= 90`：可进入因子与回测。
- `70 <= quality_score < 90`：可用于展示，但回测需显式允许。
- `quality_score < 70`：隔离并触发告警 / 补爬。
- 跨源价格差异超过阈值时，以数据更新时间、成交量完整性和来源优先级综合判定。

#### 2.4 统一数据仓库

- MySQL：元数据、财报、研报
- TSDB：分钟 / Tick 时序
- Redis：盘中实时缓存（`live:stock:{code}`）

#### 2.5 数据源契约

| 数据 | 优先来源 | 频率 | 目标存储 | 降级策略 |
|------|----------|------|----------|----------|
| 日 K | 东财 | 每日收盘后 | JSON / TSDB | 新浪兜底，失败则告警 |
| 分钟线 | 东财 | 盘中定时 | Redis / JSON | 缺失标记，不参与盘中信号 |
| 指数数据 | 东财 | 每日 / 盘中 | JSON / TSDB | 新浪兜底 |
| 财报 | 待定 | 季度 | MySQL | 人工补录 / 延迟更新 |
| 舆情 | 雪球 | 定时采集 | MySQL / JSON | 跳过并记录失败 |
| 研报 | 待定 | 每日 | MySQL | 延迟更新 |

**通用采集要求**

- 所有原始数据必须保留 `source`、`fetched_at`、`raw_version`。
- 清洗后的数据必须保留 `data_version`。
- 多源降级必须在日志和质量报告中记录，不能静默覆盖。

#### 2.6 市场范围与科创板参考模式

为降低外部数据源压力，并避免高波动科创板标的默认进入主研究流程，系统默认启用科创板参考模式：

- 科创板个股代码：`688xxx`、`689xxx`。
- 默认只保留基础行情、个股展示和科创 50 指数参考。
- 默认不进入：历史补录、质量巡检、舆情、增强数据、因子、回测、预测、实际影响、上涨候选、操作决策、模拟交易、Agent。
- 如果命令显式传入科创板代码，例如 `./run.sh predict 688001`，允许用户手动强制分析。
- 可通过环境变量关闭默认参考模式：`STAR_BOARD_REFERENCE_ONLY=0`。

**参考模式验收**

#### 4.6 因子计算规范

- 因子输入只能来自清洗、复权、质量校验后的标准数据。

- 公式或参数默认值变更时，必须生成新的因子版本。
- 上市不足 N 天、停牌、成交额过低等场景需要明确输出 `null`、跳过或降权。

```json
{
  "code": "000001",
  "trade_date": "2026-07-01",
  "horizon": "5d",
  "signal": "bullish",
  "signal_score": 72,

  "limitations": ["not_backtested", "technical_only"],

```

**信号解释**

- `signal`：`bullish`、`bearish`、`neutral` 三类方向。
- `signal_score`：0-100 分，只表示当前因子强度。

- `limitations`：必须标明是否经过回测、是否只包含技术因子。

### Phase 3：回测系统（P2）

#### 5.1 回测引擎能力

- T+1 交易规则
- 手续费 / 滑点
- 涨跌停限制
- 停牌处理
- 回测与实盘共用风控模块

#### 5.2 策略类型

- 技术策略
- 情绪策略
- 机构策略
- 产业链策略
- 多因子策略

#### 5.3 绩效指标

- 年化收益
- 最大回撤
- 夏普比率
- 胜率
- 盈亏比

#### 5.4 回测成交规则

**默认成交假设**

- 买入价：信号日后第一个可交易日开盘价。
- 卖出价：触发卖出信号后第一个可交易日开盘价。
- T+1：当日买入持仓不可当日卖出。
- 涨停：买入日涨停且无法成交时，订单取消或顺延，需由策略参数指定。
- 跌停：卖出日跌停且无法成交时，继续持有并记录未成交原因。
- 停牌：停牌期间不可买卖，价格沿用最近可交易日用于净值估算。

**交易成本默认值**

- 佣金：万分之三，最低 5 元。
- 印花税：卖出单边千分之一。
- 滑点：默认 5bp，可按策略覆盖。

#### 5.5 组合与风控规则

- 单票最大仓位：默认 10%。
- 单行业 / 单题材最大暴露：默认 30%。
- 最低流动性门槛：近 20 日日均成交额不低于指定阈值。
- 排除 ST、退市整理、新股上市不足 N 日标的。
- 最大回撤熔断：触发后停止开新仓或降仓。
- 黑名单机制：支持手动和规则生成。

#### 5.6 可验证走势预测输出

Phase 3 完成后，系统开始具备“可验证”的走势预测能力。这里的预测不是保证涨跌，而是基于历史统计和当前因子状态输出概率、置信度、收益风险比和失效条件。

```json
{
  "code": "000001",
  "trade_date": "2026-07-01",
  "horizon": "5d",
  "direction": "up",
  "probability": 0.61,
  "confidence": "medium",
  "expected_return": 0.028,
  "max_expected_drawdown": 0.045,
  "strategy_version": "trend_breakout_v1.0.0",
  "factor_versions": {
    "ma20_bias": "1.0.0",
    "rsi14": "1.0.0"
  },
  "evidence": {
    "backtest_win_rate": 0.57,
    "backtest_profit_loss_ratio": 1.35,
    "sample_count": 420
  },
  "risk_flags": ["earnings_nearby", "sector_high_volatility"],
  "updated_at": ""
}
```

**最低验收标准**

- 至少支持 `1d`、`5d`、`20d` 三类观察周期之一，MVP 优先 `5d`。
- 每个预测必须关联策略版本、因子版本和数据版本。
- 预测必须有历史样本数量，样本过少时只能输出 `low_confidence`。
- 必须同时输出方向、概率、置信度、预期收益、预期回撤和风险标记。
- 不允许输出“必涨”“必跌”等确定性结论。

#### 6.4 实盘接入前准备清单

实盘接入只能作为“实盘辅助网关”逐步推进，默认不自动下单。接入前必须准备以下内容：

| 准备项 | 必须内容 | 验收标准 |
|--------|----------|----------|
| 券商账户 | 已开通 A 股交易权限、资金账户、交易密码、风险测评 | 能通过券商官方客户端正常登录和手动交易 |
| 接口方式 | 明确使用券商官方 API、QMT/Ptrade、同花顺/通达信桥接或其他合法接口 | 有稳定 SDK、认证方式、下单/撤单/查询接口文档 |
| 行情源 | 实时行情、五档/盘口、分钟线、交易日历 | 行情延迟可观测，异常时自动停止下单 |
| 账户同步 | 资金、持仓、可用股数、冻结资金、当日成交、委托状态 | 本地账户状态与券商账户差异可检测 |
| 订单网关 | 下单、撤单、查委托、查成交、拒单原因 | 每笔订单有唯一 ID、状态机和完整审计日志 |
| 事前风控 | 单票仓位、总仓位、行业集中度、最大亏损、涨跌停、黑名单、价格偏离 | 风控未通过时不得生成可提交订单 |
| 人工确认 | 所有买入、卖出、撤单默认需要人工确认 | `requires_human_review=true` 时禁止自动提交 |
| 熔断机制 | 数据延迟、接口异常、连续亏损、回撤超限、订单失败 | 触发后自动切换只读/停止交易 |
| 密钥安全 | 交易密码、Token、证书、本地配置隔离 | 不入库、不提交 Git、只走环境变量或本地私有配置 |
| 审计留痕 | 信号、决策、风控、订单、成交、人工确认、异常 | 任意交易动作可回放到原始证据 |
| 模拟先行 | 同一策略先在 `simtrade` 跑足样本 | 模拟账户稳定、回撤可接受、执行偏差可解释 |
| 合规边界 | 明确系统仅为个人研究和辅助工具 | 禁止代客理财、绕过券商规则或规避监管要求 |

**推荐落地顺序**

1. `execution/broker_base.py`：定义券商适配器接口，不绑定具体券商。
2. `execution/paper_broker.py`：把现有模拟交易封装成 Broker，用于验证订单生命周期。
3. `risk/pre_trade.py`：实现实盘前风控检查，输出 allow/reject 和拒绝原因。
4. `execution/order_audit.py`：保存订单请求、风控结果、人工确认、券商回报和成交。
5. `execution/live_gateway.py`：只生成待确认订单，不自动提交。
6. 接入一个真实券商 Provider，先只读账户和持仓，再开放小额人工确认交易。

**实盘禁区**

- 未完成后验复盘和模拟交易稳定性验证前，不允许自动交易。
- 数据源延迟、质量低、行情缺失时，不允许下单。
- Agent 或自然语言输出不得直接触发实盘订单。
- 任意订单必须能关联到 prediction/selector/decision/review 证据包。

## 三、股票走势预测能力路径

### 3.1 预测能力分级

| 等级 | 达成阶段 | 能力描述 | 输出形式 | 是否可用于交易决策 |
|------|----------|----------|----------|--------------------|
| L0 | Phase 0 | 数据质量判断 | 数据完整性、质量分、异常标记 | 否 |
| L1 | Phase 2 | 因子倾向判断 | 看多 / 看空 / 中性信号、因子分数 | 否，只能作为观察 |
| L2 | Phase 3 | 回测验证预测 | 方向、概率、置信度、收益风险比、历史胜率 | 可用于模拟交易和人工辅助 |
| L3 | Phase 4 | 实盘验证预测 | 实时信号、订单可执行性、风控结果 | 可用于小资金实盘，但需授权 |
| L4 | Phase 5 | 解释型预测 | 预测结论、证据链、风险、失效条件、策略诊断 | 可用于投研决策，不默认自动交易 |
| L5 | Phase 6 | 产品化预测 | 看板、排名、预警、回测对比、实盘跟踪 | 可作为统一操作入口 |

### 3.2 预测闭环定义

股票走势预测必须形成以下闭环：

```text
可信数据
  → 因子信号
  → 策略规则
  → 历史回测
  → 当前预测
  → 风险过滤
  → 结果展示 / 模拟交易 / 实盘验证
  → 预测复盘
```

若缺少“历史回测”和“预测复盘”，只能称为信号提示，不能称为完整预测能力。

### 3.3 预测输出边界

**允许输出**

- 未来 `1d / 5d / 20d` 的方向倾向。
- 上涨 / 下跌概率。
- 置信度：`low`、`medium`、`high`。
- 预期收益区间和预期回撤。
- 关键驱动因子和风险因素。
- 策略历史表现和样本数量。

**禁止输出**

- 确定性涨跌结论。
- 没有数据版本、策略版本和回测证据的预测。
- 绕过风控的买卖建议。
- 由 Agent 直接触发的默认实盘下单。

### 3.4 到哪一步能完成走势预测

- **Phase 2 完成后**：可以完成“初级走势信号”，例如趋势偏多、偏空、震荡，但还不能判断该信号历史上是否赚钱。
- **Phase 3 完成后**：可以完成“可验证走势预测”，这是系统首次达到可接受的预测能力门槛。预测结果应包含方向、概率、置信度、历史胜率、收益风险比和风险标记。
- **Phase 4 完成后**：可以完成“可执行走势预测”，即预测结果能接入实时行情、交易约束和风控，但仍需要人工授权。
- **Phase 5 完成后**：可以完成“可解释走势预测”，即系统不仅输出预测，还能说明为什么、依据什么、什么情况下预测失效。
- **Phase 6 完成后**：可以完成“产品化走势预测”，在 Web 看板中展示股票池、预测排名、证据链、回测表现和实盘跟踪。

**近期最小目标**：完成 Phase 3 的单策略回测 MVP 后，即可开始对股票输出 `5d` 维度的可验证走势预测。

### 3.4.1 上涨候选池

系统需要将“可能上涨”拆成可验证的候选池排序，而不是直接给确定性结论：

```text
上涨候选分 = 预测方向/概率 + 因子分 + 趋势确认 + 回测有效性 + 实际影响 - 风险惩罚
```

**分层**

- `candidate`：上涨候选，分数达到候选线且无硬排除原因。
- `watch`：观察候选，有部分正面条件但尚未形成买入确认。
- `weak`：弱候选，条件不足。
- `rejected`：排除，预测向下、因子过低、数据质量不足或实际影响显著偏负面。

**硬排除**

- 预测方向为 `down`。
- `multi_factor_score < 45`。
- `impact_score < -20`。
- 数据质量低于最低阈值。

**候选阻断与触发条件**

- 当上涨分达到候选线，但预测未形成向上确认、趋势未站上 MA20、因子未达观察线或实际影响分为负时，最多进入 `watch`。
- `next_triggers` 必须说明进入候选池还差什么，例如“预测概率提升到 0.55 以上”“收盘价重新站上 MA20”“实际影响分修复到非负”。
- `candidate` 只表示上涨候选，不等同于买入；是否操作仍由 `decision` 和人工复核决定。

### 3.4.2 十日前历史视角滚动推演

为验证模型是否存在未来函数和过度依赖当前结果，系统支持从最近 10 个交易日前开始做逐日推演：

```text
知识截止日 T
  → 只读取 T 及以前 K 线
  → 计算当时技术因子与预测
  → 分析 T+1
  → 用 T+1 真实涨跌做复盘
  → 滚动到今天
```

**输出**

- `knowledge_cutoff`：当时认知截止日。
- `target_date`：被分析的下一交易日。
- `prediction`：当时预测方向、概率、置信度、样本数。
- `technical_signal`：当时技术信号和驱动。
- `root_cause`：基于当时可见技术因子解释 T+1 上涨或下跌的主要根因、全部观察到的根因和未命中原因。
- `learning`：汇总十日窗口中的未命中次数、看多后下跌、看空后上涨、根因分布和命中率优化思考。
- `operation_levels`：为后续单日操作细化预留的价位雏形，包括买入触发价、防守卖出价、止盈观察价。
- `actual`：下一交易日真实收益、真实方向、是否命中。

**涨跌根因分析**

根因分析先采用规则型、可审计实现，后续再接入大盘、资金、公告和产业链数据。每一步至少从以下维度解释：

| 根因维度 | 示例 | 用途 |
|----------|------|------|
| 趋势 | 站上/跌破 MA20、均线金叉/死叉 | 判断上涨或下跌是否有趋势基础 |
| 动量 | MACD 柱、20 日动量 | 判断趋势是否仍有延续性 |
| 成交量 | 放量上涨、放量下跌或分歧放大 | 判断价格变化是否有成交确认 |
| 反转风险 | RSI 超买/超卖 | 识别追高回落或超跌反弹 |
| 预测失效 | 看多后下跌、看空后上涨、边界概率误判 | 作为后续阈值校准样本 |
| 大盘环境 | 指数涨跌、市场宽度、主力/北向资金 | 判断个股涨跌是否受到系统性环境影响 |
| 个股资金 | 北向持股/净买、融资买入 | 判断资金是否支持或拖累个股方向 |
| 公告/业绩 | 业绩预告增长/下滑、二季度利润数据缺口 | 判断经营结果是否解释涨跌 |
| 产业价格事件 | 产品/材料价格维持高位、行业景气变化 | 判断利润弹性或成本压力是否解释涨跌 |

**命中率优化思考**

- 每个未命中样本必须写入 `miss_reasons`，说明错在趋势确认不足、概率边界、技术信号中性、根因不足或数据缺口。
- 外部根因必须标记 `source` 和 `source_timing`：`known_at_cutoff` 表示当时已知，`latest_context_review` 表示只用于最新复盘，不允许反向参与当时预测。
- 十日推演汇总需要输出 `false_up_count` 和 `false_down_count`，分别用于优化买入确认和卖出/减仓确认。
- 当命中率低于 50% 时，系统应降低积极操作建议，把边界概率和弱技术信号样本降级为观察。
- 当看多后下跌样本较多时，提高买入门槛：要求 MA20 站稳、放量确认、大盘风险不过高。
- 当看空后上涨样本较多时，卖出前增加超跌反弹、MACD 修复和支撑位检查。
- 后续 `review/evaluation` 需要把 replay 根因与 selector、decision 的实际收益绑定，形成可持续的阈值校准样本库。

**单日操作细化预留**

十日推演不直接生成实盘指令，但需要为后续“到达多少买/卖”保留结构化字段：

```json
{
  "operation_levels": {
    "reference_close": 10.0,
    "support_price": 9.6,
    "resistance_price": 10.8,
    "buy_trigger_price": 10.12,
    "sell_guard_price": 9.75,
    "take_profit_watch_price": 10.55,
    "note": "价位为历史推演中的操作细化雏形，需结合当日盘口和风控后才能使用"
  }
}
```

后续延伸到单日操作建议时，`decision` 需要把这些价位与当前持仓状态结合：

- 空仓：只在价格突破 `buy_trigger_price` 且根因仍成立时考虑试仓。
- 半仓：若价格跌破 `sell_guard_price` 或根因失效，考虑降仓；若突破且命中率改善，考虑加仓。
- 满仓：优先关注 `sell_guard_price` 和 `take_profit_watch_price`，避免继续追高。

**约束**

- 每一步不得读取 `target_date` 之后的数据。
- 真实收益只用于复盘，不参与当时预测。
- 默认窗口为最近 10 个交易日，可通过 `--days` 调整。

### 3.4.3 当日大盘推演与短中长线推荐

系统需要根据当日大盘环境，对股票池分别生成短线、中线、长线推荐列表，每个周期推荐 5 支股票。推荐必须结合十日前视角滚动推演，不能只依据当日静态分数。

**周期定义**

| 周期 | 观察窗口 | 核心目标 | 推荐数量 |
|------|----------|----------|----------|
| 短线 | `1d-5d` | 捕捉短期趋势延续、量价突破和事件催化 | 5 支 |
| 中线 | `5d-20d` | 捕捉趋势修复、基本面/事件兑现和资金持续性 | 5 支 |
| 长线 | `20d-60d` | 捕捉景气度、估值修复、行业趋势和回撤收益比 | 5 支 |

**大盘输入**

- 指数环境：上证指数、深证成指、创业板指、科创 50 参考、北证 50。
- 市场宽度：涨跌家数、涨停/跌停、涨幅分布。
- 资金环境：主力资金、北向/两融、行业资金流。
- 行业环境：当日强弱行业、连续强势行业、回撤行业。
- 风险环境：指数大跌、极端波动、成交额萎缩、连续亏钱效应。

**推荐评分**

```text
推荐分 = 大盘环境权重
      + 周期预测分
      + 多因子分
      + 实际影响分
      + 十日前滚动推演得分
      + 回测风险收益分
      - 风险惩罚
```

**短线推荐偏重**

- `1d/5d` 预测方向和概率。
- MA20/MA60、MACD、量能放大、短期动量。
- 当日行业/题材强度。
- 十日前推演中最近 3-5 步方向命中率和最大不利波动。

**中线推荐偏重**

- `5d/20d` 预测方向和概率。
- 多因子分、趋势修复、资金持续性。
- 业绩预告、产品价格、估值压力等实际影响。
- 十日前推演完整 10 步命中率、平均收益、回撤控制。

**长线推荐偏重**

- `20d/60d` 趋势、行业景气、估值和基本面。
- 历史最大回撤、夏普、胜率、收益风险比。
- 实际影响事件的可持续性和财报兑现周期。
- 十日前推演不要求短期方向全命中，但要求下行风险可控。

**输出结构**

```json
{
  "trade_date": "2026-07-06",
  "market_regime": "risk_off|neutral|risk_on",
  "recommendations": {
    "short": [{ "code": "", "name": "", "score": 0, "reasons": [], "risks": [], "replay": {} }],
    "medium": [{ "code": "", "name": "", "score": 0, "reasons": [], "risks": [], "replay": {} }],
    "long": [{ "code": "", "name": "", "score": 0, "reasons": [], "risks": [], "replay": {} }]
  },
  "limitations": []
}
```

**硬性约束**

- 每个周期最多推荐 5 支；不足 5 支时必须说明原因，不允许硬凑。
- 大盘处于 `risk_off` 时，短线推荐应降权，允许输出空列表或仅输出观察。
- 科创板个股默认只作为参考，不进入推荐池，除非显式关闭参考模式。
- 推荐结果必须附带十日前滚动推演摘要：命中率、覆盖率、平均收益、最大不利波动。
- 推荐不等于买入指令，仍需经过 `decision` 和人工复核。

### 3.5 指导性操作建议

在实时性低优先级、指导性高优先级的场景下，系统不应只输出“涨跌预测”，而应进一步生成单股操作建议：

```text
预测结果
  → 因子状态
  → 回测证据
  → 实际影响数据
  → Agent 解释
  → 数据质量
  → 模拟持仓
  → Decision Engine
  → 买入 / 持有 / 减仓 / 卖出 / 观望
```

**操作建议输出**

```json
{
  "code": "600378",
  "action": "watch|buy|hold|reduce|sell",
  "position_suggestion": 0.12,
  "confidence": "medium",
  "reasons": [],
  "risks": [],
  "invalid_conditions": [],
  "evidence": {
    "impact": {
      "impact_score": 0,
      "impact_direction": "neutral",
      "events": [],
      "limitations": []
    }
  },
  "requires_human_review": true
}
```

**实际影响数据要求**

- 利通电子二季度利润增长：优先使用 `20260630` 半年度正式报告或业绩预告；如果当前增强数据只有非二季度报告期，必须生成 `q2_profit_growth_gap`，并标记 `requires_q2_official_report_or_forecast`。
- 昊华科技二季度生产材料/产品价格：从公告或业绩预告原因中抽取“制冷剂产品市场价格维持高位运行、氟化工、锂电业务市场复苏”等证据，生成 `material_or_product_price` 正面事件。
- 对股票涨跌有实际影响的数据至少包括：业绩增减、利润率变化、产品/材料价格变化、估值压力、解禁压力、资金流变化、行业指数变化、舆情/公告事件。
- 事件必须输出 `event_type`、`impact_score`、`impact_direction`、`confidence`、`metrics`、`evidence`、`source`、`limitations`。
- 决策使用方式：正面影响可小幅提高目标仓位或增强观望理由；负面影响达到阈值时可阻断买入或将持有改为减仓。

**边界**

- 操作建议只用于研究、复盘和模拟交易。
- `buy / reduce / sell` 必须要求人工复核。
- 决策层不得绕过风控或直接调用真实交易接口。
- 数据质量不足、缺少预测或回测证据时，应输出 `watch` 或低置信度建议。

## 四、优秀量化系统优化蓝图

> **目标**：不推翻当前阶段计划，而是在已落地的“数据 → 因子 → 回测 → 预测”闭环上，逐步补齐优秀量化系统应有的研究、交易、风控、解释、监控和产品化能力。

### 4.1 核心能力清单

| 能力 | 优化目标 | 当前承接模块 | 后续落点 |
|------|----------|--------------|----------|
| 数据可信 | 数据完整、字段统一、可追溯、可校验 | `pipeline/`, `storage/` | 跨源 diff、数据血缘、异常补爬 |
| 因子可复用 | 因子有版本、参数、公式、输入输出规范 | `factors/` | 因子注册表、因子分布监控、因子漂移检测 |
| 策略可复现 | 固定数据、参数、版本后结果一致 | `strategy/`, `backtest/` | 策略配置化、参数快照、实验记录 |
| 回测可信 | 交易成本、T+1、涨跌停、停牌、流动性约束完整 | `backtest/`, `risk/` | 动态股票池、成交量约束、退市 / ST 处理 |
| 预测可验证 | 预测关联样本、胜率、收益风险比和复盘结果 | `prediction/` | 滚动验证、命中率统计、失效原因归因 |
| 候选可排序 | 上涨概率、因子、趋势、事件和风险形成排序 | `selector/` | 候选后验复盘、阈值自适应 |
| 事件可量化 | 财报、材料价格、解禁、估值能转成影响分 | `impact/`, `decision/` | 公告解析、产业价格源、事件复盘 |
| 风控前置 | 回测、模拟、实盘共用风控规则 | `risk/` | 账户级风控、组合敞口、行业集中度 |
| 执行可审计 | 下单、撤单、拒单、风控拦截均可回放 | `trading/` | 模拟交易、实盘网关、订单审计 |
| 组合可管理 | 支持多股票、多策略、多账户组合视角 | `portfolio/` | 现金、持仓、净值、再平衡、收益归因 |
| 结果可解释 | 输出信号来源、因子贡献、风险因素、失效条件 | `agent/`, `web/` | 选股解释、策略诊断、预测复盘报告 |
| 监控可告警 | 数据、任务、预测、交易、风控异常可告警 | `monitoring/` | 任务状态、数据延迟、订单异常、回撤告警 |

### 4.2 数据系统优化

**数据范围扩展**

- 行情：日 K、分钟线、Tick、盘口、成交明细。
- 指数：宽基指数、行业指数、主题指数。
- 基本面：财报、估值、盈利能力、成长性、资产负债。
- 公司行为：分红、送转、配股、停复牌、解禁、退市、ST。
- 资金：成交额、换手率、主力资金、北向资金、融资融券。
- 舆情：新闻、公告、雪球 / 股吧讨论、热度、情绪。
- 机构：研报、评级、目标价、覆盖机构、盈利预测。
- 产业链：题材、事件、上下游关系、价格变化。
- 宏观：利率、汇率、商品价格、流动性指标。

**优化要求**

- 区分原始层、清洗层、特征层、结果层。
- 采集失败必须支持重试、告警、降级。
- 多源数据不可静默覆盖，必须保留来源、差异和最终采用规则。
- 所有进入因子、回测、预测的数据必须通过质量门禁。

### 4.3 因子与策略优化

**因子扩展方向**

- 技术因子：趋势、动量、波动率、突破、量价背离。
- 基本面因子：估值、盈利、成长、现金流、负债。
- 情绪因子：多空比、热度、评论增速、情绪分歧。
- 机构因子：目标价空间、评级变化、覆盖机构数、盈利预测变化。
- 产业因子：事件强度、涨价强度、供需紧张度、题材热度。
- 风险因子：波动率、回撤、流动性、集中度、尾部风险。

**策略扩展方向**

- 技术趋势策略。
- 均值回归策略。
- 多因子选股策略。
- 事件驱动策略。
- 情绪反转 / 情绪动量策略。
- 行业轮动策略。
- 大盘择时策略。
- 组合再平衡策略。

**策略输出要求**

- 候选股票池。
- 买入 / 卖出 / 观望信号。
- 仓位建议。
- 触发原因。
- 风险标记。
- 策略版本和参数快照。

### 4.4 回测与预测优化

**回测增强**

- 动态股票池。
- 流动性约束。
- 成交量容量约束。
- 退市、ST、新股过滤。
- 多策略对比。
- 收益归因。
- 参数敏感性分析。
- 滚动回测和样本外验证。

**预测增强**

- 支持 `1d / 5d / 20d` 多周期预测。
- 输出方向、概率、置信度、预期收益、预期回撤。
- 输出关键驱动因子和风险因素。
- 记录策略历史表现和样本数量。
- 支持预测复盘，统计命中率、收益表现和失效原因。

**最小可用预测定义**

- 数据通过质量门禁。
- 因子无未来函数。
- 策略经过回测。
- 回测包含交易成本、T+1、停牌、涨跌停。
- 预测输出方向、概率、置信度、样本数量和风险标记。
- 预测结果支持事后复盘。

### 4.5 风控、组合与交易优化

**风控规则**

- 单票最大仓位。
- 行业 / 题材集中度限制。
- 最大回撤熔断。
- 单日亏损限制。
- 流动性过滤。
- 波动率过滤。
- ST / 退市 / 黑名单过滤。
- 重大事件窗口降仓。
- 高频交易频率限制。
- 账户级风险敞口控制。

**组合管理**

- 当前持仓、目标持仓、调仓差异。
- 现金、净值、回撤、收益曲线。
- 行业暴露、风格暴露、策略贡献。
- 组合再平衡和收益归因。

**交易系统**

- 先建设模拟交易：虚拟账户、模拟订单、模拟成交、模拟持仓。
- 再建设实盘辅助：券商接口、订单状态、成交回报、账户同步。
- 默认不允许自动下单；开启自动交易必须显式授权。
- 所有订单必须经过风控，风控拒单不可绕过。

### 4.6 AI Agent 与 Web 产品化优化

**Agent 类型**

- 数据巡检 Agent：发现缺失、异常、延迟，并触发补爬。
- 因子分析 Agent：检查因子分布、漂移、失效。
- 策略诊断 Agent：分析收益来源、回撤原因、参数敏感性。
- 选股解释 Agent：解释为什么某只股票进入股票池。
- 舆情解读 Agent：总结市场热点、情绪分歧和事件影响。
- 复盘 Agent：对预测、交易和风控结果做事后复盘。

**Web 页面**

- 市场总览。
- 自选股看板。
- 个股详情页。
- 因子排名。
- 策略信号。
- 走势预测。
- 回测报告。
- 模拟交易。
- 实盘监控。
- 风控面板。
- 数据质量面板。
- Agent 分析报告。

### 4.7 监控、审计与运维优化

**监控指标**

- 数据采集成功率。
- 数据延迟。
- 任务运行耗时。
- 回测 / 预测任务失败率。
- 实盘订单失败率。
- 风控拦截次数。
- 系统资源使用情况。

**告警场景**

- 数据源失效。
- 行情延迟。
- 质量评分过低。
- 任务连续失败。
- 回撤超限。
- 订单异常。
- 账户资金异常。

**审计要求**

- 数据版本审计。
- 策略参数审计。
- 回测结果审计。
- 预测输出审计。
- 订单和成交审计。
- Agent 操作审计。

#### 5.4.3 对标后的推荐架构

```text
Universe（股票池/参考池/过滤）
  → Signal / Insight（预测、因子、impact、replay、推荐）
  → Portfolio Construction（目标仓位、现金、行业暴露）
  → Risk Management（风控调整、拒单、降仓）
  → Execution（模拟/实盘订单）
  → Analyzer / Observer（收益、命中率、回撤、执行偏差）
  → Agent / Web Console（解释、复盘、人工确认）
```

**落地原则**

- 不推翻现有模块，而是在现有 `selector/prediction/decision/trading/replay` 上增加标准接口。
- 推荐、决策、交易必须分层：推荐只给候选；决策给操作建议；交易只执行通过风控的订单。
- 所有信号必须有有效期、来源、版本、风险和失效条件。
- 所有建议必须能被 replay/review 复盘。
- 实盘能力默认关闭，必须人工确认和风控通过。

| 盘中监控 | 4 核 CPU、8 GB RAM、Redis 可选 | 分钟线轮询、实时看板 |
| 实盘辅助 | 独立主机 / VPS、稳定网络、日志备份 | 订单、风控、审计、告警 |

### 6.3 Agent / LLM 运行需求

**接入托管 LLM API**

- 不需要本地 GPU。
- 需要稳定网络和 API Key。
- 需要单独配置密钥，不得写入代码仓库。
- 需要请求预算和调用频率限制。
- 所有 LLM 输出必须落审计日志，不能直接触发实盘交易。

**接入本地 LLM**

- 仅用于离线解释和研究，不建议作为实盘路径首选。
- 7B 级模型建议 16-32 GB 内存或 8 GB+ 显存。
- 14B 及以上模型建议 32-64 GB 内存或更高显存。
- 本地模型输出仍必须经过规则校验和风控过滤。

### 6.4 配置与密钥要求

- 数据源、数据库、Redis、LLM、券商接口密钥必须通过环境变量或本地配置注入。
- `.env`、密钥文件、券商证书不得提交到仓库。
- 实盘交易配置必须与研究配置隔离。
- 生产环境需要保留任务日志、预测日志、Agent 审计日志和订单日志。

## 七、Agent 接入方案

### 7.1 当前 Agent 状态

当前系统已具备规则型 Agent 与可替换 Provider 框架：

- `agent/context.py`：读取股票、因子、信号、舆情、增强数据、预测、回测和质量报告。
- `agent/schemas.py`：生成 `EvidencePackage`，限制 Agent 只能消费结构化证据包。
- `agent/providers/`：定义 Provider 抽象，当前支持 `rule` 与未配置时降级的 `llm` 占位。
- `agent/policy.py`：拦截越权输出、无证据结论和绕过人工确认的交易动作。
- `agent/audit.py`：保存输入摘要、输出摘要、模型、提示词版本、权限校验和降级原因。
- `agent/stock_explainer.py`：解释选股依据。
- `agent/strategy_diagnosis.py`：诊断策略表现。
- `agent/predict_review.py`：复核预测方向、证据和风险。
- `agent/orchestrator.py`：编排并输出 Agent 报告。
- `tasks/agent_job.py`：批量生成 Agent 报告，支持 `--provider rule|llm`。

当前默认不调用外部 LLM；选择 `--provider llm` 时如未配置真实 Provider，会自动降级为规则型 Agent 并写入审计记录。Agent 不自动下单，所有输出仅用于解释、总结和研究建议。

### 7.2 外部 Agent 接入目标

- 自动总结个股机会和风险。
- 解释预测结论背后的证据链。
- 诊断策略失效原因。
- 生成每日复盘和任务异常摘要。
- 给出参数优化建议。
- 辅助生成研究报告。

### 7.3 接入架构

```text
量化数据层
  → Agent Context Builder
  → Evidence Package（证据包）
  → LLM / 外部 Agent Provider
  → Policy Guard（权限与合规校验）
  → Agent Report
  → Web 看板 / 审计日志 / 人工确认
```

**建议新增模块**

```text
quant_system/
  agent/
    providers/
      base.py
      rule.py
      llm_stub.py
    prompts/
      stock_review.md
      strategy_review.md
      daily_brief.md
    policy.py
    schemas.py
    audit.py
```

### 7.4 Agent 输入证据包

外部 Agent 不应直接自由读取全仓库数据，而应消费结构化证据包：

```json
{
  "code": "600378",
  "trade_date": "2026-07-06",
  "stock": {},
  "factors": {},
  "signal": {},
  "prediction": {},
  "backtest": {},
  "quality": {},
  "sentiment": {},
  "enhance": {},
  "risk_limits": {},
  "allowed_actions": ["analyze", "summarize", "suggest"]
}
```

**输入要求**

- 只提供与任务相关的数据。
- 附带数据版本、因子版本、策略版本。
- 明确 `allowed_actions`。
- 缺失数据必须显式标记，禁止让 Agent 猜测。

### 7.5 Agent 输出格式

```json
{
  "summary": "",
  "direction_view": "bullish|bearish|neutral",
  "confidence": "low|medium|high",
  "evidence": [],
  "risks": [],
  "failure_conditions": [],
  "suggested_actions": [],
  "forbidden_actions": [],
  "requires_human_review": true
}
```

**输出要求**

- 必须引用证据包字段。
- 必须说明风险和失效条件。
- 不能输出确定性涨跌结论。
- 不能绕过风控给出下单指令。
- 涉及实盘动作时必须 `requires_human_review = true`。

### 7.6 权限与安全边界

| 权限等级 | 允许能力 | 禁止能力 |
|----------|----------|----------|
| L0 只读分析 | 读取证据包、生成解释 | 修改数据、下单 |
| L1 研究建议 | 生成策略建议、参数建议 | 自动改生产参数 |
| L2 模拟辅助 | 生成模拟交易建议 | 实盘下单 |
| L3 实盘辅助 | 生成待确认订单建议 | 绕过人工确认和风控 |
| L4 自动交易 | 仅在显式授权后可用 | 默认禁用 |

当前阶段只建议开放 L0-L1；模拟交易完成后再考虑 L2。

### 7.7 接入步骤

1. ✅ 固化 `EvidencePackage` 和标准 Provider 输出结构。
2. ✅ 增加 `agent/providers/base.py` 抽象接口。
3. ⏳ 接入一个托管 LLM Provider，密钥走环境变量。
4. ✅ 增加 `agent/policy.py`，校验输出是否越权。
5. ✅ 增加 `agent/audit.py`，保存输入、输出、模型、时间、版本。
6. ✅ 在 `agent_job` 中增加 `--provider rule|llm` 参数，默认仍使用 `rule`。
7. ✅ 将 Provider 输出接入 `reports/agent/`，并保留规则型 Agent 作为兜底。
8. ✅ 增加测试：缺字段、越权输出、LLM 未配置降级。

### 7.8 验收标准

- 无 API Key 时系统仍可使用规则型 Agent。
- LLM 调用失败时自动降级为规则型 Agent。
- Agent 输出必须包含证据、风险、限制和免责声明。
- 所有 Agent 输入输出可审计。
- Agent 不得直接修改数据、策略参数或触发交易。

## 八、通用约束与风险声明

### 8.1 工程约束

- 所有关键流程必须可重跑、可追溯、可审计。
- 策略、因子、数据版本需要在回测报告和实盘日志中记录。
- 任何数据源失败、降级、补爬、字段异常都必须进入日志。
- 回测和实盘应尽量共享成交约束、风控规则和标的过滤规则。

### 8.2 投资与合规风险

- 系统输出不构成投资建议，仅作为研究和辅助决策工具。
- 回测收益不代表未来收益。
- 数据源异常、延迟或字段变更可能导致信号失真。
- 实盘交易必须由用户主动授权，并保留人工干预和熔断机制。
- 涉及第三方数据采集时，需要遵守对应平台的服务条款和数据使用边界。
